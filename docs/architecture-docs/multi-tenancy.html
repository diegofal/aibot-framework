Multi-Tenancy & BaaS

The framework supports a full shared-infrastructure multi-tenant mode for running Bots-as-a-Service (BaaS). Gated by config.multiTenant.enabled.

Two Levels of Isolation

User (within bot)End-users of a single botuserIsolation.enabled flag, user_id column in core_memory, per-user daily logs, per-user memory search

Authentication & Authorization

Dual Auth Model

The system supports two token types for different use cases:

Token TypePrefixUse CaseLifetime
Session tokensess_Dashboard login (email + password)24 hours (in-memory, auto-expiry)
API keyaibot_Programmatic / external integrationsPersistent (stored on tenant record)

Session & Credential Stores

  • SessionStore (src/tenant/session-store.ts) — In-memory map of active sessions with 24h TTL and periodic cleanup. Creates sess_-prefixed tokens via crypto.randomBytes.
  • AdminCredentialStore (src/tenant/admin-credentials.ts) — File-based admin credentials (data/admin-credentials.json). Passwords hashed with argon2id via Bun.password.hash. Supports first-run setup (create-once) and verification.
  • passwordHash field on Tenant — Tenants have an optional passwordHash field for email+password login. Also hashed with argon2id.

Three Auth Tiers

TierMiddlewareRoutes
PublicNonePOST /api/tenants, GET /api/status, GET /api/auth/status, POST /api/auth/login, POST /api/auth/admin-setup, POST /api/webhooks/stripe, POST /api/onboarding/signup
TenantcreateTenantAuthMiddleware (Bearer token — sess_ or aibot_)/api/agents/*, /api/conversations/*, /api/billing/*, etc.
AdmincreateAdminAuthMiddleware (session token with admin role, or ADMIN_API_KEY env)/api/admin/*

The tenant auth middleware resolves the Bearer token in two ways: if the token starts with sess_, it looks up the SessionStore; otherwise it matches against tenant API keys (aibot_ prefix) or the ADMIN_API_KEY env var. Admin sessions (role admin) get the virtual __admin__ tenantId with enterprise plan.

Route Scoping

All route handlers use getTenantId(c), scopeBots(), and isBotAccessible() from src/tenant/tenant-scoping.ts to filter data per-tenant. When tenantId is __admin__, both helpers treat it as a super-tenant: scopeBots returns all bots (including legacy bots without tenantId), and isBotAccessible returns true for any bot.

Resolved by resolveTenantPaths() in src/tenant/tenant-paths.ts.

Path Sandboxing

isPathWithinTenant(path, tenantRoot) validates that file operations stay within the tenant’s root. _tenantRoot is injected into tool args by ToolExecutor.

Collaboration Boundaries

AgentRegistry.listOtherAgents(excludeBotId, tenantId) only returns agents from the same tenant, preventing cross-tenant bot-to-bot communication.

Tenant Configuration

Config Merge Order

global → tenant → bot (bot wins). Implemented in resolveAgentConfigWithTenant().

BYOK (Bring Your Own Key)

Tenants can provide their own API keys for Claude, ElevenLabs, Brave Search via TenantConfigStore.

Schema

TenantConfigSchema (Zod) in src/tenant/tenant-config.ts covers: LLM backend, model, API keys, conversation defaults, feature toggles, branding.

Enforcement Points

  • ConversationPipeline: Pre-LLM check blocks messages if quota exceeded; post-reply records usage.
  • AgentLoop: Pre-cycle check skips execution if API call quota exceeded; post-cycle records LLM + tool usage.
  • REST API: createQuotaCheckMiddleware returns 429; createRateLimitMiddleware enforces per-minute sliding window.

Graceful Degradation

At 80% usage: X-Quota-Warning: approaching. At 90%: X-Quota-Warning: critical. At 100%: requests blocked with upgrade URL.

Dashboard Authentication

When multiTenant.enabled = true, the web dashboard requires authentication via email + password. In single-tenant mode, no auth is required — fully backward compatible.

Flow

  1. app.js fetches /api/auth/status (public) to check multiTenantEnabled and adminSetupRequired
  2. If adminSetupRequired = true (first run, no admin account yet) → admin setup form shown
  3. Admin setup POSTs to /api/auth/admin-setup with { email, password } (min 8 chars). Only works once.
  4. If multi-tenant enabled and no session token in sessionStorage → sidebar hidden, login page rendered
  5. Login form POSTs to /api/auth/login with { email, password }
  6. Server checks admin credentials (argon2id) first, then tenant by email. Returns { sessionToken, role, name }
  7. On success: session token (sess_ prefix), role, name stored in sessionStorage. All api() calls add Authorization: Bearer <token>
  8. On 401: token cleared, auth:required event fires → redirect to login
  9. Logout POSTs to /api/auth/logout, which invalidates the session server-side

Session Token in Middlewares

Both tenant and admin auth middlewares accept sess_-prefixed session tokens. The token is resolved via SessionStore: admin-role sessions get __admin__ tenantId with enterprise plan; tenant-role sessions resolve to the tenant record. API keys (aibot_ prefix) and the ADMIN_API_KEY env var remain supported for programmatic access.

WebSocket Auth

Activity and log WebSocket connections pass the token as ?token=... query parameter. The server validates the token (session or API key) before upgrade in multi-tenant mode.
