Dynamic Tools

Tools are OpenAI-compatible function definitions that the LLM can invoke during conversations and agent loop cycles. They are registered in ToolRegistry (src/bot/tool-registry.ts) and executed by the ToolExecutor. Each tool has a JSON schema for parameters, safety validations, and per-bot enable/disable controls.

Skills are higher-level capabilities packaged as src/skills/<id>/ directories with a skill.json manifest and an index.ts entry point. Skills can register Telegram commands, cron jobs, and tool integrations.

File & Exec

5 tools

file_read

Read the contents of a file with line numbers. Validates paths against a base directory, blocks sensitive file patterns (.env, credentials, SSH keys), and rejects symlinks escaping the allowed directory.
Parameters
  • path string required — File path relative to the allowed base directory
  • offset number — Line number to start reading from (1-based)
  • limit number — Maximum number of lines to return

file_write

Write content to a file. Creates the file if it doesn't exist, or overwrites it. Set append=true to append instead of overwriting. Creates intermediate directories automatically. Same security validations as file_read.
Parameters
  • path string required — File path relative to the allowed base directory
  • content string required — The content to write
  • append boolean — Append instead of overwrite (default: false)

file_edit

Edit an existing file by replacing exact text. Searches for old_text and replaces with new_text. Fails if old_text is not found or appears multiple times (unless replace_all is true). Preferred over file_write for modifications.
Parameters
  • path string required — File path relative to the allowed base directory
  • old_text string required — The exact text to find
  • new_text string required — The replacement text
  • replace_all boolean — Replace all occurrences (default: false)

exec

Execute a shell command on the host machine and return stdout/stderr. Includes safety checks blocking dangerous commands (rm -rf /, fork bombs, mkfs, etc.), configurable allow/deny patterns, and optional background mode that returns a session ID for the process tool.
Parameters
  • command string required — The shell command to execute
  • workdir string — Working directory for the command
  • background boolean — Run in background, returns session ID (default: false)

process

Manage background processes started by exec with background=true. List running/finished processes, poll new output, send stdin input, kill running processes, or clear finished sessions from the registry.
Parameters
  • action enum required — list | poll | write | kill | clear
  • session_id string — Session ID (required for poll, write, kill, clear)
  • input string — Text to send to stdin (for "write" action)

Ask the human operator a question. Non-blocking: the question is queued to the web inbox and optionally sent as a Telegram notification. The answer will be injected into the bot's next agent loop cycle. Continue working on other tasks in the meantime.
Parameters
  • question string required — The question to ask the human operator
  • timeout_minutes number — Minutes to wait for response (default: 30, max: 120)

ask_permission

Request human approval before performing a sensitive action. Non-blocking: the request is queued to the web permissions page. The decision (approved/denied) will be injected into the bot's next agent loop cycle. Continue working on other tasks in the meantime.

History lifecycle: pending → decided → consumed → executed/failed. After approval/denial, the agent loop consumes the decision and reports execution outcome (summary, tool calls, success/failure) back to the store. History is bounded (100 entries, 24h TTL).
API endpoints: GET /api/ask-permission/history?limit=N (recent decisions), GET /api/ask-permission/history/:id (single entry for status polling).
Parameters
  • action string required — The action the bot wants to perform
  • resource string required — The resource or target of the action
  • description string required — Human-readable explanation of why permission is needed
  • urgency string — Urgency level: low | normal | high (default: normal)
  • timeout_minutes number — Minutes to wait for decision (default: 30, max: 120)

phone_call

Make phone calls via Twilio or manage a phonebook of contacts. Supports calling by contact name or direct phone number, adding/removing contacts, and listing the phonebook. Messages are spoken using TTS with configurable language and voice.
Parameters
  • action string required — call | add_contact | list_contacts | remove_contact
  • contact string — Contact name or direct phone number
  • message string — Message to speak in the call
  • phone_number string — Phone number in E.164 format (for add_contact)
  • loop number — Times to repeat the message (default: 1, use 3 for emergencies)

Dynamic Tools

Beyond the 27 built-in tools, agents can create custom tools at runtime using the create_tool tool. Dynamic tools go through a human approval workflow:

  1. The agent creates a tool via create_tool (TypeScript or shell command)
  2. Static analysis checks for dangerous patterns (eval, child_process, fs, etc.)
  3. The tool is stored in DynamicToolStore with status PENDING
  4. A human reviews and approves/rejects via the web dashboard
  5. Approved tools are loaded by DynamicToolRegistry and become available to agents

Dynamic tools support per-bot scoping — a tool can be available to all bots or restricted to the bot that created it.

Tool Architecture

Tool Interface

Every tool implements the Tool interface from src/tools/types.ts:

interface Tool {
  definition: ToolDefinition;  // OpenAI-compatible function schema
  execute(args: Record<string, unknown>, logger: Logger): Promise<ToolResult>;
}

interface ToolResult {
  success: boolean;
  content: string;
}

interface ToolDefinition {
  type: 'function';
  function: {
    name: string;
    description: string;
    parameters: { type: 'object'; properties: Record<string, unknown>; required?: string[] };
  };
  outputSchema?: ZodType;   // Optional structured output validation
  maxRetries?: number;       // Auto-retry on transient failures
}

Registration Flow

ToolRegistry.initializeAll() checks each tool's config flag and creates instances. The registry provides filtered views per bot (getDefinitionsForBot(), getToolsForBot()) that respect per-bot disabledTools and dynamic tool scope exclusions.

Tool Categories & Pre-Selection

Tools are organized into 10 categories (TOOL_CATEGORIES in tool-registry.ts): web, memory, soul, files, system, social, calendar, communication, browser, production.

During agent loop execution, the planner selects which categories the executor needs (via toolCategories in its JSON response). The agent loop then calls getDefinitionsByCategories() to send only the matching tools, reducing token usage by ~2,500–3,500 tokens per call. Three tools (get_datetime, ask_human, ask_permission) are always included. External/dynamic tools (not in any category) also pass through unconditionally.

Config: agentLoop.toolPreSelection (default: true). Set to false to disable and always send all tools.

Execution Pipeline

The ToolExecutor handles the full lifecycle: argument injection (_botId, _chatId), disabled tool rejection, execution with logging, and result formatting. External content is wrapped with <<<EXTERNAL_UNTRUSTED_CONTENT>>> markers so the LLM knows to treat it as untrusted.

Safety Layers

Layer Mechanism
Per-bot disable config.bots[].disabledTools array
SSRF protection Blocked host patterns in web_fetch and browser
Path traversal file_read/write/edit validate against basePath + symlink checks
Command safety exec blocks fork bombs, rm -rf /, mkfs, shutdown, etc.
Sensitive files Built-in deny patterns for .env, credentials, SSH keys, etc.

Collaboration safety collaborate/delegate_to_bot excluded from collaboration-mode tool sets
External content wrapExternalContent() markers prevent prompt injection

External Skills

External skill packages follow the TSC format: a directory containing a skill.json manifest and an index.ts handler file. They are loaded at startup from directories listed in config.skillsFolders.paths[].

Loading Pipeline

Settings UI (folder picker) → Config (skillsFolders.paths[])
                                         ↓
Startup: scan all folders → Loader (discover + parse + import) → Adapter (ToolHandler → Tool) → ToolRegistry (ctx.tools[])
                                         ↓
Agent Edit UI → Per-bot skill enable/disable (BotConfig.disabledSkills[])

Key Files

FileRole
src/core/external-skill-loader.ts Discovers skill dirs, reads manifests, validates, dynamically imports handlers
src/core/external-tool-adapter.ts Adapts TSC ToolHandler signature to framework Tool interface with namespaced names
src/bot/tool-registry.ts initializeExternalSkills() orchestrates loading; getExcludedSet() handles disabledSkills

Tool Namespacing

Each external tool is namespaced as ${skillId}_${toolName} (e.g. github_repo_list, linear_issue_create). This prevents name collisions with built-in tools and between skills.

Per-Bot Filtering

BotConfig.disabledSkills: ["github"] disables ALL tools from the github skill for that bot. Individual tools can still be disabled via the existing disabledTools mechanism. Both sets are merged in ToolRegistry.getExcludedSet().

Skills Dashboard

The web UI includes a unified Skills page at #/skills that merges built-in and external skills into a single browsable list. Built-in skills are read-only; external skills support full CRUD operations.

Security

All write operations (create, update, delete, apply) validate that the target directory is within one of the configured skillsFolders.paths[] using resolve() + startsWith(). Requests targeting directories outside the allowed paths are rejected with 403 Forbidden.

AI Generation

src/skill-generator.ts follows the same pattern as src/soul-generator.ts: builds a prompt with format specs and 1–2 examples from existing external skills, calls claudeGenerate(), strips markdown fences, and parses the JSON response into { skillJson, handlerCode }. The generate endpoint returns a preview; the apply endpoint writes files to disk.

Integration Pattern

Each integration follows the same three-layer pattern:

  1. Native Tool (src/tools/<service>.ts) — LLM-callable tool with typed parameters, registered in ToolRegistry
  2. Built-in Skill (src/skills/<service>/) — Telegram command handlers with skill.json manifest
  3. External Skill (productions/) — TSC-format package loadable via skillsFolders config
