▶ Tenant Data Isolation (Physical) 5 stories · 2 phases Audit Complete

Soul files and productions are correctly scoped per-tenant under data/tenants/{tid}/bots/{botId}/. However, sessions, conversations, and shared JSON stores remain in global directories with only logical isolation (by globally-unique botId). This works correctly today but prevents per-tenant data export/deletion and creates bot ID namespace pressure across tenants. resolveTenantPaths() in src/tenant/tenant-paths.ts already computes tenant-scoped session paths but nothing uses them.

Phase 1 — Physical path isolation (3 stories)
    ▶ SessionManager — tenant-scoped session storage

    Currently SessionManager (src/session.ts) is instantiated once globally with config.session.dataDir. All tenants share data/sessions/, sessions.json metadata, and active-conversations.json.

    Options
    • A: Per-tenant SessionManager instances (cleanest isolation, higher memory)
    • B: Single instance with tenant-prefixed paths (less change, shared metadata file)
    Current state

    resolveTenantPaths().sessionsDir already returns data/tenants/{tid}/bots/{botId}/sessions/ but is unused. clearBotSessions(botId) works via prefix matching — no data leak, just co-location.

    Files
    • src/session.ts — constructor, initialize(), transcript paths
    • src/index.ts — SessionManager instantiation
    • src/bot/bot-manager.ts — passes SessionManager to BotContext
    ▶ ConversationsService — tenant-scoped conversation storage

    ConversationsService uses a single config.conversations.baseDir (data/conversations/). All tenants' conversations are stored in data/conversations/{botId}/.

    Impact
    • Web routes filter via scopeBots() (application-level), not filesystem
    • Export/import (bot-export-service.ts) reads from global path
    • Bot reset (deleteAllForBot) works by botId directory — correct but co-located
    Files
    • src/conversations/service.ts — botDir(), conversationsPath()
    • src/bot/bot-manager.ts:141 — instantiation
    • src/bot/bot-export-service.ts — export/import paths
    ▶ Shared JSON stores — tenant-scoped persistence paths

    Several stores persist to global dataDir paths shared across tenants:

    • ask-human/answered.json — AskHumanStore (src/bot/ask-human-store.ts)
    • ask-permission/history.json — AskPermissionStore (src/bot/ask-permission-store.ts)
    • collaboration/tracker.json — CollaborationTracker (src/collaboration-tracker.ts)
    • collaboration/sessions.json — CollaborationSessionManager (src/collaboration-session.ts)
    • schedules.json — AgentScheduler (src/bot/agent-scheduler.ts)

    All are internally scoped by botId (which is globally unique), so no data leaks. All clearForBot() methods persist to disk after clearing.

    Note

    Lower priority than sessions/conversations. These are small operational stores. Moving them only matters for per-tenant backup/restore or compliance deletion.

Phase 2 — Namespace & compliance (2 stories)
    ▶ Bot ID namespacing — auto-prefix with tenantId

    Bot IDs are currently globally unique (agents.ts:79 rejects duplicates across all tenants). Tenant A creating bot "support" blocks Tenant B from the same name.

    Proposed
    • Auto-prefix tenant bot IDs: {tenantId}:{botName} or {tenantId}__{botName}
    • Uniqueness check scoped to tenant, not global
    • Display name remains unqualified in UI
    Impact

    Touches all botId references: session keys, file paths, config persistence, web routes, Telegram bot registration. Breaking change — requires migration for existing tenant bots.

    ▶ Per-tenant data export & deletion

    Enable exporting or deleting ALL data for a tenant (GDPR-style). Requires physical isolation from Phase 1.

    • DELETE /api/tenants/:id/data — remove all tenant data
    • GET /api/tenants/:id/export — download tenant data archive
    • Enumerate all data stores, collect by tenant, package or purge
