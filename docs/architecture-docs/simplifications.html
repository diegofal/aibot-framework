7 Config Schema Monolith Low

All Zod validation schemas live in a single 503-line file, making it hard to locate the schema for a specific feature.

Current State
src/config.ts contains 38 Zod schemas in 503 lines, covering everything from bot configuration to web tools, memory settings, collaboration options, session management, and karma. Finding a specific schema requires scrolling through the entire file. Schema interdependencies are implicit via ordering.
Proposed Solution
Split into a src/config/schemas/ directory with one file per logical group:
  • bot.schema.ts — BotConfig, conversation overrides, agent loop
  • tools.schema.ts — web tools, exec, file, browser, datetime
  • memory.schema.ts — auto-RAG, memory search, memory flush, session
  • infra.schema.ts — paths, logging, ollama, web, cron
  • features.schema.ts — collaboration, soul, karma, productions, humanizer
A barrel src/config/index.ts re-exports the composed ConfigSchema.
Expected Benefit
Each schema file is ~80–120 lines. Schema ownership becomes clear. Adding a new feature schema does not require editing a 500-line file.
Estimated Effort
Medium — mechanical extraction but requires updating all import paths. No logic changes, just file reorganization.
Files Affected
  • src/config.ts

8 SystemPromptBuilder Concatenation Low

The system prompt is built by concatenating 15+ private methods, making it difficult to test individual prompt sections or compose them dynamically.

Current State
system-prompt-builder.ts uses ~15 private methods, each returning a string block. The main build() method concatenates them all with newlines. Testing requires calling build() and searching the full output for expected substrings. There is no way to test or reuse individual sections, and adding conditional blocks requires if-statements scattered across the concatenation chain.
Proposed Solution
Adopt a registered block pattern: each section is a named block with a { id, priority, condition?, render() } signature. The builder collects blocks, filters by condition, sorts by priority, and joins. Individual blocks can be unit-tested by calling render() directly. Plugins or skills could register custom prompt blocks.
Expected Benefit
Individual prompt sections become unit-testable. Conditional inclusion becomes declarative. Opens the door for skill-contributed prompt blocks.
Estimated Effort
Medium — requires defining the block interface and migrating each private method to a block registration. The actual prompt content does not change.
Files Affected
  • src/bot/system-prompt-builder.ts

9 Collaboration Flow Duplication Low

Three collaboration flows duplicate prompt construction and context assembly logic.

Current State
collaboration.ts implements three distinct flows — visible messaging, internal messaging, and delegation — each of which independently constructs the LLM prompt by assembling the target bot's persona, conversation history, and collaboration instructions. The assembly logic is ~70% identical across flows, differing only in which instructions and response format are appended.
Proposed Solution
Extract a shared buildCollaborationContext(targetBot, flow, history) function that assembles the common prompt prefix (persona + history + base instructions), then each flow appends its specific suffix. This follows the same pattern used by SystemPromptBuilder for conversation vs. collaboration modes.
Expected Benefit
Eliminates ~100 lines of duplicated prompt assembly. Changes to shared context (e.g., adding token counts) apply to all flows at once.
Estimated Effort
Small — the common prefix is easy to identify and extract. The three flows already follow a similar structure.
Files Affected
  • src/bot/collaboration.ts

10 Memory Module Tight Coupling Low

Eight memory module files are tightly coupled through a shared raw Database object, making it hard to test or swap storage backends.

Current State
The src/memory/ directory contains 8 files that all receive and operate on a raw Database object directly. Each module constructs its own SQL queries, manages its own table schemas, and handles serialization. There is no shared abstraction for common operations like "search by embedding" or "insert with timestamp". Testing requires a real SQLite database because the raw Database API is not easily mockable.
Proposed Solution
Introduce a MemoryDB abstraction layer that encapsulates the raw database and provides typed methods for common operations:
  • search(table, embedding, limit) — vector similarity search
  • insert(table, record) — typed insert with automatic timestamps
  • query(table, filters) — filtered reads with pagination
Memory modules would depend on MemoryDB instead of the raw Database, making them testable with a mock implementation.
Expected Benefit
Enables unit testing memory modules without SQLite. Makes it feasible to swap storage backends (e.g., PostgreSQL for production). Centralizes SQL query construction and error handling.
Estimated Effort
Large — requires defining the abstraction, migrating all 8 modules, and ensuring no performance regressions in query patterns. Should be done incrementally.
Files Affected
  • src/memory/*.ts
