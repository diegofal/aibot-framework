Architecture Overview

How the bot core is structured: facade pattern, focused modules, shared context, and dependency ordering.

Facade Pattern — BotManager

The entire bot core is exposed through a single public API: BotManager. It lives in src/bot/bot-manager.ts (627 lines) and is the only export from src/bot/index.ts. External callers — the web dashboard, the CLI entry point, tests — never touch internal modules directly.

BotManager is a slim facade. It does not contain business logic itself; instead, it composes focused modules and delegates to them. Its responsibilities are limited to:

  • Constructor — creates the shared BotContext, instantiates all modules in dependency order, and wires lazy callbacks for circular dependencies.
  • startBot / stopBot / stopAll — lifecycle management for Telegram polling and headless agents.
  • sendMessage — delivers text to a Telegram chat via a specific bot instance.
  • Agent loop delegates — startAgentLoop, runAgentLoop, getAgentLoopState, wakeAgentLoop.
  • Ask-human delegates — getAskHumanPending, answerAskHuman, dismissAskHuman.
  • Collaboration delegates — sendVisibleMessage, handleDelegation, discoverAgents, collaborationStep.

Because the facade is thin, adding a new feature means creating or modifying a focused module and then exposing it through one or two delegate methods on BotManager.

Composition Patterns

Shared BotContext (mutable state by reference)

Every module receives the same BotContext object (defined in types.ts, 69 lines). It holds references to all infrastructure services (SessionManager, MemoryManager, SkillRegistry, etc.) plus per-bot mutable state like bots: Map<string, Bot>, runningBots: Set<string>, and activeModels: Map<string, string>.

Because BotContext is passed by reference, when BotManager.startBot() adds a bot to runningBots, all modules immediately see the change — no event bus, no pub/sub, no syncing. 

// types.ts — BotContext (abridged)
export interface BotContext {
  readonly config: Config;
  readonly sessionManager: SessionManager;
  readonly skillRegistry: SkillRegistry;
  readonly memoryManager: MemoryManager | undefined;
  readonly logger: Logger;

  // Per-bot mutable state (shared by reference)
  readonly bots: Map<string, Bot>;
  readonly runningBots: Set<string>;
  readonly activeModels: Map<string, string>;
  readonly tools: Tool[];
  readonly toolDefinitions: ToolDefinition[];

  // Helper methods
  getActiveModel(botId: string): string;
  getLLMClient(botId: string): LLMClient;
  getSoulLoader(botId: string): SoulLoader;
}

Lazy callbacks for circular dependencies

The collaboration and delegation tools need a reference to CollaborationManager, but CollaborationManager itself is created after ToolRegistry. This circular dependency is broken with lazy callbacks: 

// bot-manager.ts constructor — wire circular deps
this.toolRegistry.initializeAll(
  () => this.collaborationManager,   // lazy — not evaluated at init time
  () => ({
    discoverAgents: (excludeBotId) =>
      this.collaborationManager.discoverAgents(excludeBotId),
    collaborationStep: (sessionId, targetBotId, message, sourceBotId) =>
      this.collaborationManager.collaborationStep(
        sessionId, targetBotId, message, sourceBotId
      ),
    // ...
  }),
);

The arrow function () => this.collaborationManager captures this in a closure but is not invoked until a tool is actually called at runtime — by which point collaborationManager is fully initialized.

Initialization order matters

Modules are instantiated in dependency order inside the BotManager constructor. The sequence reflects the dependency graph shown above: 

// bot-manager.ts constructor — initialization order
this.toolRegistry       = new ToolRegistry(ctx);                            // 1. no deps
this.systemPromptBuilder = new SystemPromptBuilder(ctx, this.toolRegistry); // 2. reads ToolRegistry
this.memoryFlusher      = new MemoryFlusher(ctx);                           // 3. no deps
this.groupActivation    = new GroupActivation(ctx);                         // 4. no deps
this.conversationPipeline = new ConversationPipeline(                       // 5. uses 1, 2, 3
  ctx, this.systemPromptBuilder, this.memoryFlusher, this.toolRegistry
);
this.collaborationManager = new CollaborationManager(                       // 6. uses 1, 2
  ctx, this.systemPromptBuilder, this.toolRegistry
);
this.handlerRegistrar   = new HandlerRegistrar(                             // 7. uses 4, 5, 3, 1
  ctx, this.conversationPipeline, this.groupActivation,
  this.memoryFlusher, this.toolRegistry, this.askHumanStore
);
this.agentLoop          = new AgentLoop(                                    // 8. uses 1, 2
  ctx, this.systemPromptBuilder, this.toolRegistry
);

If a new module depends on an existing one, it must be instantiated after its dependency in this sequence. If it would create a circular dependency, use the lazy callback pattern described above.
