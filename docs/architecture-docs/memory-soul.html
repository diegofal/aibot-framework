Memory & Soul System

4-layer memory architecture with RAG-powered retrieval

4-Layer Memory Architecture

The memory system is organized into four distinct layers, each serving a different purpose in the bot's ability to remember and retrieve information. Higher layers are volatile and fast; lower layers are persistent and deep.

Memory Modules

The memory system lives in src/memory/ with 9 focused modules coordinated by MemoryManager.

Memory Lifecycle

A conversation message travels through the full memory pipeline, from volatile session storage to persistent vector-indexed knowledge.

Lifecycle Steps

  1. Conversation — User sends messages, bot replies. All messages stored as ChatMessage[] in session memory.
  2. Session Expiry — When session timeout elapses or user runs /clear, MemoryFlusher.flushSessionToMemory() is called.
  3. LLM Scoring — The conversation transcript is sent to the LLM (Claude or Ollama) with a prompt to extract facts with importance scores (1-10) and categories.
  4. Dual Storage — Facts are written to Core Memory (SQLite key-value) and/or daily memory log files (memory/YYYY-MM-DD.md).
  5. Indexing — A file watcher (debounced) detects new/changed .md files, chunks them by heading boundaries and token targets, and generates embeddings.
  6. Vector Store — Chunks and embeddings are stored in SQLite with FTS5 full-text search and auto-sync triggers for keyword matching.

RAG Search Flow

The hybridSearch() function in search.ts combines vector similarity with keyword matching for robust retrieval. Results are merged with Core Memory entries weighted by importance. botId is required on all search paths — vector, keyword, and core memory queries all filter by file path prefix ({botId}/ or sessions/bot-{botId}-) and core memory filters by bot_id column, enforcing per-bot isolation with no fallback.

Vector Search Path

Query text is embedded via Ollama using the configured embedding model. The query embedding is compared against all stored chunk embeddings using cosine similarity. Results scored 0-1.

queryEmbedding = embeddingService.getEmbedding(query)
for each chunk:
  score = cosineSimilarity(queryEmbedding, chunk.embedding)

Keyword Search Path

Query is tokenized, stop-words removed (bilingual: Spanish + English), and tokens with prefix matching are fed to SQLite FTS5. BM25 ranking scores are normalized to 0-1.

tokens = query.split().filter(stopWords).map(t => t + '*')
ftsQuery = tokens.join(' OR ')
score = 1 / (1 + |bm25_rank|)

Merge and Scoring

The final score is a weighted combination of both search paths, controlled by config.vectorWeight and config.keywordWeight:

finalScore = vectorWeight * vectorScore + keywordWeight * keywordScore

Results are filtered by minScore, sorted descending, and limited to maxResults.

Temporal Decay (Enabled by Default)

After merging vector + keyword scores, an exponential temporal decay is applied to boost recent memories. The decay uses each chunk's last_indexed_at timestamp from the files table:

decayMultiplier = (1 - weight) + weight * exp(-ageDays * ln2 / halfLifeDays)

With defaults (halfLifeDays: 30, weight: 0.3), a brand-new memory gets full score (1.0x), a 30-day-old memory gets 0.85x, and very old memories floor at 0.7x. Configurable via soul.search.temporalDecay (enabled, halfLifeDays, weight).

MMR Diversity Re-ranking (Opt-in)

After temporal decay and score-based ranking, an optional Maximal Marginal Relevance (MMR) step can re-rank results to reduce near-duplicate chunks. Enabled via soul.search.mmr.enabled: true in config.

MMR iteratively selects items that maximize: λ × relevance − (1−λ) × max_similarity_to_selected. Text similarity is computed using Jaccard distance on tokenized content. The lambda parameter (default 0.7) controls the balance: higher values favor relevance, lower values favor diversity.

Pipeline: vector + keyword → score merge → temporal decay → build results → MMR → core memory merge → return

Core Memory entries with importance >= 7 are prioritized above regular results. Lower-importance core entries fill remaining slots.

Stop Words

Bilingual filtering removes common words in both Spanish and English before FTS5 queries. Only tokens with 3+ characters pass through.

Soul System

Each bot has a soul/ directory containing persistent identity files that define personality, motivations, and long-term memory. These files are indexed for RAG search and reviewed by the soul health check system.

IDENTITY.md

Core identity definition: who the bot is, its name, description, and fundamental characteristics.

SOUL.md

Personality traits, communication style, tone of voice, and behavioral patterns.

MOTIVATIONS.md

Inner motivations, impulses, and drives that shape behavior. Updated through reflection cycles.

GOALS.md

Current objectives and focus areas. Can be updated by the bot itself or by operators.

MEMORY.md

Consolidated long-term memory. Merged from daily logs by the memory consolidation process.

memory/ directory

Daily memory log files (YYYY-MM-DD.md). Archived into memory/archive/ after consolidation.

Soul Health Check

The soul-health-check.ts module runs on startup (with cooldown) and performs three operations:

  1. Structural Lint (no LLM) — Validates required files exist (IDENTITY.md, SOUL.md, MOTIVATIONS.md), checks for duplicated sections leaked into SOUL.md, detects stale placeholders in MOTIVATIONS.md, and verifies the memory/ directory exists.
  2. Memory Consolidation — Merges old daily logs (older than today) into MEMORY.md using Claude CLI. Deduplicates facts, removes noise (agent-loop artifacts), organizes by natural categories, and archives processed logs.
  3. Quality Review — Spawns Claude CLI with --allowedTools Read,Edit,Write to review soul files for personality drift, vague language, contradictions, and outdated content. Makes targeted, minimal edits preserving voice.

Consolidation and quality review run concurrently since they operate on different files. A cooldown timestamp (.last-health-check) prevents redundant runs.

Per-Bot Isolation in Skill Cron Jobs

Skill cron jobs (reflection, calibrate, improve) are registered per-bot in the CronService. Each job's CronPayload carries a botId field. When the cron service fires a skill job, resolveSkillHandler in src/index.ts looks up the bot config, calls resolveAgentConfig() to determine the per-bot soulDir, and injects it into the SkillContext. Skills never fall back to a shared root directory — if soulDir is not set, they error and skip execution. The improve skill uses a per-bot concurrency lock (Set<string>) instead of a global boolean, allowing concurrent runs for different bots.

Core Memory

Core Memory is a MemGPT-style structured key-value store in SQLite. It enables the bot to maintain explicit, categorized knowledge about itself and the people it interacts with. High-importance entries are automatically injected into the system prompt. Entries are scoped per-bot via a bot_id column — each bot can only read and write its own core memory, and clearCoreMemoryForBot() only deletes entries for the target bot.

Categories

Category Description Example
identity Who the bot is (name, description, values, style) name: "Lina"
relationships Data about specific users the bot interacts with diego_timezone: "UTC-3"
preferences Bot's own preferences and behavioral choices reply_style: "concise"
goals Long-term objectives and focus areas learn_cooking: "..."
constraints Self-imposed limits and behavioral boundaries no_politics: "..."

API Methods

Method Description
get(category, key) Retrieve a single entry by category and key. Returns CoreMemoryEntry | null.
set(category, key, value, importance?) Insert or update an entry. Key max 100 chars, value max 2000 chars, importance 1-10 (default 5).
delete(category, key) Delete an entry. Returns boolean indicating whether a row was removed.
search(query, category?, limit?) Search entries by query string matching key or value. Optionally filtered by category.
list(category?, minImportance?) List entries optionally filtered by category and/or minimum importance threshold.
renderForSystemPrompt(maxChars?) Render entries with importance >= 5 as formatted markdown for system prompt injection (default 800 chars max).

System Prompt Injection

renderForSystemPrompt() queries all entries with importance >= 5, groups them by category, and formats up to 5 items per category within the character budget. The output is injected as a ## Core Memory section in the system prompt, giving the bot persistent awareness of its most important knowledge.

## Core Memory

**Identity**:
- name: Lina
- description: A creative and empathetic AI companion

**Relationships**:
- diego_preference: prefers technical discussions
- maria_language: speaks in Spanish

Memory Consolidation

The soul-health-check.ts module's consolidateMemory() function merges daily memory logs into a unified MEMORY.md file. This prevents unbounded growth of daily log files and keeps the bot's long-term memory organized.

Consolidation Process

Key Details

  • getUnconsolidatedLogs() finds all YYYY-MM-DD.md files in memory/ older than today (excluding legacy.md).
  • Existing MEMORY.md is read first; falls back to legacy.md for first-time migration.
  • Before overwriting, MEMORY.md is backed up via backupSoulFile() into .versions/.
  • Claude CLI runs in prompt-only mode (-p flag, --output-format text) with no file editing tools.
  • Processed daily logs are moved to memory/archive/ via renameSync().
  • Output starts with a <!-- last-consolidated: YYYY-MM-DD --> metadata comment.

SQLite Schema Reference

All persistent memory data lives in a single SQLite database configured via config.memorySearch.dbPath. Uses WAL journaling and foreign keys.

Tables

files

Tracks indexed files with content hashes for incremental re-indexing.

id INTEGER PRIMARY KEY
path TEXT UNIQUE NOT NULL
content_hash TEXT NOT NULL
last_indexed_at TEXT NOT NULL
chunk_count INTEGER DEFAULT 0
source_type TEXT DEFAULT 'memory'

chunks

Text chunks with embeddings. Foreign-keyed to files with cascading deletes.

id INTEGER PRIMARY KEY
file_id INTEGER REFERENCES files(id)
chunk_index INTEGER NOT NULL
content TEXT NOT NULL
start_line / end_line INTEGER
token_estimate INTEGER
content_hash TEXT NOT NULL
embedding BLOB

embedding_cache

Caches generated embeddings by content hash to avoid redundant Ollama calls.

content_hash TEXT PRIMARY KEY
embedding BLOB NOT NULL
model TEXT NOT NULL
created_at TEXT DEFAULT now()

core_memory

Structured key-value store with importance scoring, category indexing, and per-bot isolation via bot_id.

id INTEGER PRIMARY KEY
bot_id TEXT NOT NULL DEFAULT 'default'
category TEXT NOT NULL
key TEXT NOT NULL
value TEXT NOT NULL
importance INTEGER DEFAULT 5
created_at / updated_at TEXT
UNIQUE(bot_id, category, key)

FTS5 Virtual Table

A chunks_fts virtual table using SQLite FTS5 is kept in sync with the chunks table via AFTER INSERT, AFTER DELETE, and AFTER UPDATE triggers. This enables BM25-ranked keyword search alongside vector similarity.
