Agent Loop

Autonomous agent execution: planning, strategy, and tool use

Overview

The agent loop is the autonomous execution engine of the AIBot Framework. It runs independently of Telegram messages, allowing bots to pursue goals, execute plans, and take actions proactively.

The loop was refactored from a single monolithic agent-loop.ts (1,482 lines) into a family of focused modules. The main orchestrator agent-loop.ts (724 lines) delegates to: agent-scheduler.ts (scheduling and concurrency), agent-retry-engine.ts (retry logic), agent-planner.ts (plan generation), agent-strategist.ts (strategic oversight), agent-loop-utils.ts (shared utilities), and llm-json-parser.ts (generic JSON parsing). Supporting modules include agent-loop-prompts.ts (prompt templates), agent-feedback-store.ts (human feedback persistence), ask-human-store.ts (human-in-the-loop questions), ask-permission-store.ts (human-in-the-loop permission requests), and tool-executor.ts (centralized tool execution engine).

Phase Diagram

Each agent loop cycle follows this sequence of phases. The strategist runs conditionally every N cycles.

Operating Modes

The agent loop supports two distinct operating modes, configurable per bot via botConfig.agentLoop.mode.

Periodic Mode Default

Runs at configurable intervals (e.g., every 5 minutes). The bot sleeps between runs using an interruptible sleep mechanism that can be woken via wakeUp().

Key Config

everyInterval duration string (e.g. "5m", "1h"). Parsed by parseDurationMs().
maxDurationMsHard timeout per cycle. A session timeout guard triggers at 80% of this limit.

Behavior

Each bot gets its own async loop via runBotLoop(). After execution, sleep duration is computed from config. On transient errors (timeout, network), exponential-backoff retries are attempted before falling back to the normal 5-min+ sleep cycle.

Continuous Mode Advanced

Runs back-to-back cycles with only a short configurable pause between them. The bot stays active as long as it has work to do.

Key Config

modeSet to "continuous" on the per-bot agentLoop config.
continuousPauseMsMilliseconds to wait between back-to-back cycles (default: 5000).
continuousMemoryEveryLog to memory every N cycles to avoid flooding (default: 5).

Tracking

Tracks continuousCycleCount and sessionStartAt per bot schedule. Uses a separate continuous planner prompt that includes the last cycle summary for context continuity.

Strategist

The strategist is the strategic oversight layer, implemented in agent-strategist.ts. It runs conditionally every N cycles (configured via strategist.everyCycles) and only when the minimum interval has elapsed (strategist.minInterval). Both conditions must be met (AND logic).

What It Does

Reviews goals and recent activity (last 7 days), performs goal operations, sets focus for upcoming cycles, and provides strategic reflection.

Goal Operations

The strategist can perform four types of goal operations:

Action Description Fields
add Add a new goal to active list goal, priority?, notes?
complete Move goal from active to completed goal (substring match), outcome?
update Modify an existing active goal goal (substring match), status?, priority?, notes?
remove Delete a goal from active list goal (substring match)

Output Format: StrategistResult

interface StrategistResult {
  goal_operations: GoalOperation[];
  focus: string;           // 1-2 sentence directive for next cycles
  reflection: string;      // Analysis of agent's current state
  next_strategy_in?: string; // Suggested next run, e.g. "6h"
}

Retry Behavior

The strategist uses runStrategist() with up to 1 retry. First attempt uses temperature 0.4, retry uses temperature 0. JSON parsing is handled by the shared parseLLMJson<T> utility from llm-json-parser.ts. If both attempts fail to produce parseable JSON, the strategist result is null and the cycle continues without strategic guidance.

Planner

Planner logic lives in agent-planner.ts. Each cycle, the planner makes a lightweight LLM call to decide whether to act and what steps to take. It receives the full bot context: identity, soul, motivations, goals, available tools, recent memory, feedback, and the strategist's focus. A unified runPlannerWithRetry() handles both periodic and continuous modes.

}

Priority Levels

PriorityMeaning
highUrgent or time-sensitive tasks, direct goal progress
mediumRoutine work, exploration, incremental progress
lowCreative exploration, boundary-pushing, novel research
noneNothing novel to do. Empty plan required. Triggers idle tracking.

Novelty Imperative

The planner prompt includes a strong Novelty Imperative: repeating the same action is treated as worse than doing nothing. A recentActionsDigest is built from the bot's last 24 hours of actions, flagging patterns repeated 2+ or 3+ times. Anti-patterns such as "reviewing goals just to review them" or "saving reflections to memory" are explicitly banned.

Retry Behavior

Uses runPlannerWithRetry() (from agent-planner.ts) with 1 retry. First attempt at temperature 0.3, retry at 0. JSON parsing uses the shared parseLLMJson<T> from llm-json-parser.ts. If parsing still fails, returns an idle plan (priority: 'none') rather than generating busywork.

Executor

The executor takes the planner's steps and carries them out using the full LLM tool-calling interface. It builds a system prompt via SystemPromptBuilder in autonomous mode and sends the plan as a user message via buildExecutorPrompt().

File Tree Scanning

Before building the executor prompt, scanFileTree() (from agent-loop-utils.ts) walks the bot's workDir (max depth 3, max 100 entries) and passes the result as fileTree to buildExecutorPrompt(). When the directory is empty or missing, the prompt explicitly tells the LLM that the directory is empty and to use file_write only — preventing blind file_read/file_edit attempts on non-existent files.

When productions.enabled === false for a bot, the file tree scan is skipped entirely and the executor prompt omits the "Working Directory Contents", "Production Directory Rules", and file-centric tool usage instructions. This saves context tokens and prevents the LLM from being pushed toward file production for bots that should focus on conversations, research, or goal management. The productionsEnabled flag is passed via ExecutorPromptInput.

Proportional Karma Penalties

Tool errors are scored proportionally per tool name: if >50% of a tool's calls in a cycle failed, delta is -1; if ≤50% failed, delta is 0 (logged as Minor: for visibility but no score impact). Each tool is evaluated independently.

Execution Flow

The AgentLoop creates a ToolExecutor instance per cycle, then calls executor.createCallback() to get a function compatible with LLMClient.chat(). The LLM makes tool calls, which the callback dispatches through the executor. After the LLM conversation finishes, executor.getExecutionLog() retrieves all recorded tool calls for the cycle result.

ToolLoopDetector

src/bot/tool-loop-detector.ts
Detects when an LLM is stuck in repetitive tool call patterns and provides warning/blocking signals to break the loop. Adapted from OpenClaw's tool-loop-detection.ts with architectural changes to fit the ToolExecutor class pattern.

4 Detection Strategies

DetectorPatternLevels
global_circuit_breaker Same tool+args+result repeated past absolute ceiling Critical only (default: 25)
known_poll_no_progress Configurable "poll" tools returning identical results Warning (8) / Critical (16)
ping_pong A↔B alternating call patterns with no progress Warning (8) / Critical (16, requires no-progress evidence)
generic_repeat Same tool+args repeated (result-aware: skips if results differ) Warning only (8)

Integration Flow

The AgentLoop creates a ToolLoopDetector from the agentLoop.loopDetection config (merged global + per-bot overrides) and passes it to ToolExecutor via options. Inside execute():

  1. Pre-check: detector.check(name, args) before execution. Critical → block and return error. Warning → capture message.
  2. Record: detector.recordCall(name, args) after tool runs.
  3. Outcome: detector.recordOutcome(name, args, result) with content hash.
  4. Inject: If warning was captured, append [LOOP WARNING: ...] to result so the LLM sees it in context.

Configuration

// config.json → agentLoop.loopDetection
{
  "enabled": true,            // default: true
  "historySize": 30,          // sliding window
  "warningThreshold": 8,
  "criticalThreshold": 16,
  "globalCircuitBreakerThreshold": 25,
  "detectors": {
    "genericRepeat": true,
    "knownPollNoProgress": true,
    "pingPong": true
  },
  "knownPollTools": [
    { "toolName": "process", "actions": ["poll", "log", "list"] }
  ]
}

User Awareness

The agent loop can be configured to scan session metadata and inject a summary of recently active users into the planner prompt. This enables coach bots and other proactive agents to see who is active and send messages to them.

Configuration

config.json → agentLoop.userAwareness
{
  "enabled": false,            // default: false
  "activeWindowHours": 24,     // look back window (1-168)
  "maxUsers": 5                // max users in summary (1-20)
}

Per-bot overrides via bots[].agentLoop.userAwareness (same fields, all optional).

How It Works

  1. buildActiveUsersSummary() scans SessionManager.listSessions() for keys matching bot:{botId}:private:*.
  2. Filters by lastActivityAt within the configured time window.
  3. Sorts by most recent, takes top N users.
  4. Returns a markdown table injected into the planner system prompt.

src/tools/send-proactive-message.ts

The send_proactive_message tool (category: communication) allows the agent loop to send messages to users. For numeric chat IDs, it sends via Telegram bot.api.sendMessage(). For non-numeric IDs (widget users), it appends to the session transcript, visible on reconnect via the history endpoint.

Feedback System

The AgentFeedbackStore allows humans to submit feedback to bots via the web dashboard. Feedback is processed at the beginning of each agent loop cycle, before the strategist and planner run.

AgentFeedbackStore

src/bot/agent-feedback-store.ts
Manages agent-level feedback with JSONL persistence in the bot's soul directory.

Feedback Interface

interface AgentFeedback {
  id: string;          // UUID
  botId: string;
  content: string;     // The feedback text
  createdAt: string;   // ISO timestamp
  status: 'pending' | 'applied' | 'dismissed';
  appliedAt?: string;  // When processed
  response?: string;   // Bot's response to the feedback
}

Lifecycle

StepDescription
1. SubmitHuman submits feedback via web dashboard. Appended to {soulDir}/feedback.jsonl.
2. DetectAt cycle start, getPending(botId) checks for unprocessed feedback.
3. ProcessEach feedback item gets a dedicated LLM call with soul-modifying tools only (manage_goals, update_soul, update_identity, save_memory). Uses Claude CLI directly.
4. ApplyFeedback marked as applied with the bot's response text. Soul files re-read.

Storage

Persisted as JSONL (one JSON object per line) in {soulDir}/feedback.jsonl. Loaded into memory on startup via loadFromDisk(). On status changes, the entire file is rewritten via rewriteJSONL().

Ask-Human Flow

The AskHumanStore enables agents to request human input during autonomous operation. When an agent cannot proceed without a decision, it uses the ask_human tool to pose a question.

Flow

StepDescription
1. AskAgent calls ask_human tool. AskHumanStore.ask() registers the question and returns a promise.
2. DisplayQuestion appears in the web dashboard inbox and optionally as a Telegram message.
3. AnswerHuman answers via web (answerById()) or Telegram reply (handleReply()). Promise resolves.
4. ConsumeNext agent loop cycle calls consumeAnswersForBot(). Answered questions are injected into the planner context as top priority.
5. TimeoutUnanswered questions auto-reject after timeoutMs. Timer cleared on answer or dismiss.

Reply Matching

When a Telegram reply arrives, handleReply() first tries to match by replyToMessageId. If no reply-to but exactly one pending question exists in that chat, it matches automatically. Answered questions are stored in a separate map and consumed by the agent loop on the next cycle.

Dismissal

Questions can be dismissed via dismissById(), which rejects the promise with an error. The agent sees this as a timeout and moves on to other work. Pending questions are also listed in the planner prompt with instructions not to re-ask.

Ask-Permission Flow

The AskPermissionStore enables agents to request human approval before performing sensitive actions during autonomous operation. Unlike ask_human (free-text answers), permission requests receive binary decisions: approved or denied.

AskPermissionStore

Persistence

Pending, resolved, and history entries are all persisted to data/ask-permission/history.json. On restart, pending requests are restored with fresh deferred promises. This means permission requests survive dispose() and process restarts.

Allowed Write Paths

Each bot can configure allowedWritePaths (default: ["productions/"]). Writes to these paths do not require ask_permission. The planner prompt explicitly lists which paths are allowed and which require permission.

Flow

StepDescription
1. RequestAgent calls ask_permission tool. AskPermissionStore.request() registers the request, persists to disk, and returns a promise.
2. DisplayPermission request appears in the web dashboard permissions page with action, resource, description, and urgency.
3. DecideHuman approves (approveById()) or denies (denyById()). Promise resolves with the decision.
4. ConsumeNext agent loop cycle calls consumeDecisionsForBot(). Permission decisions are injected into the planner context.

Immediate Run & Pending Wake

requestImmediateRun(botId) triggers an agent loop cycle as soon as possible for a specific bot. This is called automatically after every user conversation completes (via the MessageBuffer callback in BotManager), ensuring the bot acts on user requests without waiting for the next scheduled cycle.

Behavior

Bot StateAction
Idle (not executing)Sets schedule.nextRunAt = Date.now() and calls wakeUp() to abort the current sleep, causing the bot loop to run immediately.
Busy (executing)Adds botId to a pendingWakeRequests Set. When the current cycle finishes, runBotLoop() checks and consumes the pending wake, skipping the inter-cycle sleep.

Design Decisions

  • Set, not queue: Multiple requests for the same bot coalesce into one pending wake. The planner sees all recent conversation context on re-run.
  • Fire-and-forget: The conversation pipeline does not await the agent loop. Response latency is unaffected.
  • No LLM classification: Every user message triggers a wake. The planner already decides if there's actionable work (returns priority: none cheaply if not).

Graceful Stop (Draining)

AgentScheduler.gracefulStop(timeoutMs?) enables safe shutdown: it sets a draining flag, wakes all sleeping bots, and waits for executing cycles to finish (or timeout). The runBotLoop() and runLoop() guards check !this.draining, so sleeping bots exit immediately while executing bots finish their current cycle before exiting. After all loops settle, stop() runs for full cleanup.

State Exposure

Standing Directives

Operators can assign standing directives to individual bots — ongoing behavioral instructions that are injected into strategist, planner, and executor prompts via a dedicated "Operator Directives" section. Unlike goals (which the agent manages), directives are operator-controlled and persistent.

Config fields (BotAgentLoopOverrideSchema):

  • directives: string[] — custom free-text directives (max 10, 500 chars each)
  • presetDirectives: string[] — IDs of built-in directive bundles (e.g. "conversation-review")

resolveDirectives(botConfig) in agent-scheduler.ts merges custom directives with expanded preset definitions from PRESET_DIRECTIVE_DEFINITIONS in agent-loop-prompts.ts. The result is passed to all prompt builders and rendered by buildDirectivesSection().

API: GET /api/agents/:id/directives returns directives, presetDirectives, resolvedDirectives (expanded), and availablePresets.

Retry on Error

Retry logic lives in agent-retry-engine.ts. When executeSingleBot() returns an error, executeSingleBotWithRetry() checks if the error is transient (timeout, network, rate limit) via isRetryableError(). Permanent errors (auth, permission) skip retry entirely. Each retry gets a fresh maxDurationMs budget.

Retry Config

// agentLoop.retry (global or per-bot override)
{
  maxRetries: 2,          // default: 2 (3 total attempts)
  initialDelayMs: 10000,  // default: 10s
  maxDelayMs: 60000,      // default: 60s cap
  backoffMultiplier: 2    // default: exponential x2
}

Backoff Formula

delay = min(initialDelayMs * multiplier^attempt, maxDelayMs) ± 20% jitter. The jitter prevents thundering herd when multiple bots fail simultaneously.

Side-Effect Suppression

During intermediate retries, logToMemory() and sendReport() are suppressed via the suppressSideEffects option. Only the final attempt (success or last failure) writes to memory and sends reports.

Schedule Tracking

BotSchedule tracks retryCount and lastErrorMessage. Both reset to zero on success. The dashboard displays retry count in the bot schedules table and shows a retry attempt badge in the last results table.
