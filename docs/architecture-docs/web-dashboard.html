Web Dashboard

REST API, WebSocket streaming, and SPA frontend

Architecture

The web server is built on Bun's native HTTP server with Hono as the routing framework. It serves two distinct concerns from a single process:

  • REST API — All routes are mounted under /api/* and organized into focused route modules (one file per domain).
  • SPA Frontend — Static files are served from the web/ directory via Hono's serveStatic middleware, with a fallback to index.html for client-side routing.
  • WebSocket — A dedicated /ws/logs endpoint streams real-time log entries to connected dashboard clients.

The server entry point is src/web/server.ts. It receives all major services as a WebServerDeps dependency bag: 

export type WebServerDeps = {
  config: Config;
  configPath: string;
  logger: Logger;
  botManager: BotManager;
  sessionManager: SessionManager;
  skillRegistry: SkillRegistry;
  cronService: CronService;
};

Some route groups are conditionally mounted based on whether their backing service is enabled in the configuration:

  • /api/productions — only if ProductionsService is available.
  • /api/karma — only if KarmaService is available.
  • /api/tools — only if DynamicToolStore and DynamicToolRegistry are available.
  • /api/agent-proposals — only if AgentProposalStore is available (agentProposals.enabled: true).

Productions Service Features

  • Auto-numbered filenames — New files get 01_, 02_, etc. prefixes per directory for chronological ordering.
  • Self-contained index.html SPA — Auto-generated HTML page with sidebar navigation, file viewer (markdown rendered via marked.js, HTML in sandboxed iframe), active goals from GOALS.md, and stats. Matches architecture-docs dark theme.
  • Static file serving — GET /productions-view/:botId/* serves production files with correct MIME types, enabling the index.html SPA to fetch files via relative URLs.
  • Rich file descriptions — Extracts "Title -- First sentence" from file content (120 char cap).
  • Coherence checking — Heuristic checks (template ratio, min content size, heading/paragraph balance) without LLM.
  • Archive from dashboard — File viewer has Archive button with reason modal, moves files to archived/.

WebSocket — Log & Activity Streaming

The server provides two WebSocket endpoints: /ws/logs for structured log streaming and /ws/activity for real-time bot activity events. Both use Bun.serve's upgrade mechanism with discriminated data.type to route connections to separate client sets. The log streaming works as follows:

Connection lifecycle

  1. Upgrade — Bun.serve's fetch handler intercepts requests to /ws/logs and upgrades them to WebSocket connections. All other requests are delegated to the Hono app.
  2. History — On connect, the last 100 lines of the log file are parsed from disk and sent as a { type: "history", lines: [...] } message so the client can render a backlog immediately.
  3. Live streaming — The server uses fs.watch() on the log file (configurable via config.logging.file, defaults to ./data/logs/aibot.log). When new bytes are appended, they are read using a tracked file offset, parsed as JSON-lines, and broadcast to all connected clients as { type: "logs", lines: [...] }.
  4. Rotation handling — If the file size shrinks (log rotation or truncation), the offset resets to 0 so the new file contents are picked up from the beginning.
  5. Cleanup — Closed WebSocket connections are removed from the client set. Failed sends also trigger client removal.
// WebSocket message types
{ type: "history", lines: [...] }   // sent on connect (last 100 entries)
{ type: "logs",    lines: [...] }   // sent on file change (new entries)

// Each line is a parsed JSON log object, e.g.:
{
  "level": 30,
  "time": 1708617600000,
  "msg": "Agent loop completed",
  "botId": "poet",
  "module": "agent-loop"
}

Both WebSocket endpoints are unidirectional — the server does not process any client-to-server messages. The message handler is a no-op.

Activity Stream (/ws/activity)

The activity WebSocket streams events from the ActivityStream module (src/bot/activity-stream.ts). On connect, the last 50 events are sent as { type: "history", events: [...] }. New events are broadcast as { type: "activity", event: {...} }. The ActivityStream is an EventEmitter with a circular buffer (default 2000 events) that receives events from agent-loop, conversation-pipeline, memory-flush, and collaboration modules. The getSlice(limit, offset) method supports paginated access for infinite scroll. REST endpoints: GET /api/activity?limit=&offset= (events) and GET /api/logs?limit=&offset= (system logs) return { events/lines, total }. 

// Activity event types
tool:start | tool:end | tool:error    // Tool execution lifecycle
llm:start | llm:end                   // LLM call lifecycle (include backend + caller fields)
llm:error | llm:fallback              // LLM failure and primary→fallback transitions
agent:phase | agent:idle | agent:result // Agent loop phases
memory:flush | memory:rag             // Memory operations
collab:start | collab:end             // Bot-to-bot collaboration
compaction                            // Context compaction

// ActivityEvent interface
{ type, botId, timestamp, phase?, data? }

// LLM event data fields (llm:start/end/error)
{ backend: 'ollama'|'claude-cli', caller: 'conversation'|'planner'|'strategist'|'executor'|'feedback', durationMs? }

SPA Frontend Pages

The frontend is a single-page application served from the web/ directory. Hono's serveStatic middleware serves assets, with a wildcard fallback to index.html for client-side routing. The dashboard includes the following pages:

Dashboard

Overview statistics: uptime, running bots, active sessions, pending inbox items, and cron job status at a glance.

Activity

Real-time activity stream powered by the /ws/activity WebSocket. Shows everything the bot does as it does it: tool calls (start/end/error), LLM requests, agent loop phases (strategist/planner/executor), memory operations (flush, RAG), and collaboration sessions. Color-coded event type badges, filter by bot or event type, expandable event details. 2000-event DOM limit with auto-scroll, pause/resume, and infinite scroll (scroll up to load older entries via GET /api/activity). System Logs tab also supports scroll-up pagination via GET /api/logs (5000-line DOM limit).

Inbox

Ask-human question queue. Bots can use the ask_human tool to request operator input. Questions are listed with answer and dismiss actions. Badge polling via /api/ask-human/count. Messages may include file attachments shown as clickable chips below the message bubble; clicking a chip opens a modal preview fetched from /api/files/:botId/*path.

Permissions

Permission request queue. Bots can use the ask_permission tool to request approval before performing sensitive actions. Requests are listed with approve and deny actions, showing action, resource, description, and urgency. Badge polling via /api/ask-permission/count.

Agents

Bot management: create, edit, start, stop, reset, clone, and delete agents. Per-agent configuration overrides for model, conversation parameters, and agent loop settings. Soul file generation and application.

Sessions

Conversation history browser. Lists all active sessions with metadata (bot, chat, message count, timestamps). Click through to view full transcripts with pagination.

Cron Jobs

Scheduled task management: create, edit, enable/disable, force-run, and delete cron jobs. View run history with timestamps and results. Supports selective log entry deletion.

Tools

Dynamic tool inventory. Lists bot-created tools with their source code, parameters, and approval status. Approve, reject, edit metadata, or delete tools from the dashboard.

Tool Runner

Direct tool execution page. Lists ALL tools (built-in + dynamic) in a searchable sidebar grouped by source. Select a tool to view its description and auto-generated parameter form (string, number, boolean, enum, object/array). Execute directly without LLM mediation and view results with success/failure badge and duration.

Productions

Creative production tracking. Lists each bot's outputs (files, text, etc.) with evaluation workflow: approve/reject with 1-5 ratings and written feedback. AI-generated summaries of recent work via Claude CLI.

Conversations

Direct web-based chat with bots. Three types: general (discuss goals, motivations, behavior), productions (discuss recent work), and inbox (follow-up on bot questions). Uses the same unified conversation pipeline as Telegram: SystemPromptBuilder.build() for full system prompts (soul, memory, core memory, RAG, humanizer, karma, tool blocks) and prefetchMemoryContext() for automatic RAG context injection. Multi-turn message history sent as proper ChatMessage[] array.

Feedback

Agent feedback management. Submit, view, or auto-generate harsh quality audits via Claude CLI that analyze soul coherence, production quality, goal progress, and behavioral patterns. Feedback entries can be applied or dismissed.

Karma

Bot karma scoring system. View current scores for all bots, browse paginated event history, and make manual adjustments with reasons. Tracks automatic karma events from production evaluations and other signals.

Agent Proposals

Review agent creation proposals submitted by bots via create_agent. Pending proposals show as cards with full details (name, role, personality, skills, justification). Approve to trigger agent creation pipeline (config entry + soul directory + AI soul generation), or reject with optional note. History section shows resolved proposals. Badge in sidebar shows pending count.

Activity (unified)

Unified real-time monitoring page with two tabs:

  • Events — Structured bot events (tool calls, LLM, agent phases, memory, collaboration) via /ws/activity. Filter by agent and event type.
  • System Logs — Pino log output (debug/info/warn/error) via /ws/logs. Filter by level, agent, and free-text search.

Both WebSockets run in parallel so neither tab loses events while viewing the other. Pause/Resume and Clear act on the active tab. The old #/logs route redirects to #/activity?tab=logs.

Integrations

Ollama diagnostic page. Shows connection status, latency, configured timeout, and available models. Includes a test chat form that proxies messages through the same OllamaClient code path the bots use, displaying the response and duration.

Settings

Configuration editor for session, collaboration, and memory search settings. Edits are persisted directly to config.json. Covers session reset policies, group activation, LLM relevance checks, memory search (MMR diversity re-ranking, Auto-RAG), and collaboration parameters (max rounds, cooldown, timeouts).

Skills

Read-only inventory of registered skills. Displays skill metadata (id, name, version, description), registered commands, scheduled jobs, and LLM backend assignment.

Authentication

In single-tenant mode (default), the dashboard operates without authentication. All API endpoints and WebSocket connections are openly accessible. This assumes a trusted local network or an external reverse proxy handling auth.

In multi-tenant mode (multiTenant.enabled = true), the dashboard enforces username/password authentication with session tokens:

  • app.js fetches GET /api/auth/status at boot to check multiTenantEnabled and adminSetupRequired flags
  • On first run (adminSetupRequired = true), a setup form is shown. The admin submits email + password (min 8 chars) via POST /api/auth/admin-setup. Credentials are stored with argon2id hashing. This endpoint only works once.
  • If multi-tenant is enabled and no session token exists in sessionStorage, the sidebar is hidden and the login page is shown
  • Users authenticate via POST /api/auth/login with { email, password }. The server checks admin credentials first, then tenant credentials by email.
  • On success, a session token (sess_ prefix, 24h TTL) is returned and stored in sessionStorage. All api() calls send it as Authorization: Bearer <token>
  • On 401, the token is cleared and the user is redirected back to login
  • Logout calls POST /api/auth/logout which invalidates the session server-side via SessionStore
  • WebSocket connections pass the token as a ?token= query parameter
