tool-executor.ts

src/bot/tool-executor.ts · 729 lines

Centralized tool execution engine that replaces duplicated logic across AgentLoop, ToolRegistry, and CollaborationManager. Handles tool lookup, disabled tool filtering, context injection (_chatId, _botId), per-bot workDir path resolution, output validation with Zod schemas, retry with exponential backoff, productions logging with quality gate, and observable lifecycle events (tool:start, tool:end, tool:error). Extends EventEmitter. Error paths are consolidated via buildFailResult() helper that unifies 6 different error response construction sites. Auto-bridges tool lifecycle events to ctx.activityStream in the constructor (when available), ensuring all ToolExecutor instances — agent-loop, conversation pipeline, collaboration — surface events in the Activity stream without manual wiring.

ask-permission-store.ts

src/bot/ask-permission-store.ts

Manages pending "ask_permission" requests. When an agent needs human approval before performing a sensitive action, it registers a permission request with action, resource, description, and urgency. Permissions never expire — they persist until the human decides. Pending requests are persisted to disk and survive restarts. Decisions (approved/denied) are held for consumption by the next agent loop cycle.

The slim facade that composes all bot modules and exposes the public API. Constructs the shared BotContext, initializes modules in dependency order, wires lazy callbacks for circular dependencies, and delegates all operations to the appropriate module. Manages bot lifecycle (start/stop/reset), Telegram polling with 409-conflict retry, headless mode, and all delegate methods for agent loop, ask-human, ask-permission, feedback, collaboration, dynamic tools, productions, and karma. State Persistence

Disk-backed stores

Persistence Pattern

All in-memory stores that hold important state now persist to config.paths.data (typically ./data/). Each store accepts an optional dataDir parameter. When provided, the store loads state on construction and writes to disk on every mutation. Pending entries containing Promises are NOT persisted and are treated as expired on restart.

New Module: tool-audit-log.ts

src/bot/tool-audit-log.ts — Persists tool execution audit entries to daily JSONL files. Subscribes to tool:end events from ToolExecutor in the agent loop. Each entry records timestamp, botId, chatId, toolName, args, success, result (truncated to 500 chars), durationMs, and retryAttempts.

New Module: activity-stream.ts — LLM Diagnostics

src/bot/activity-stream.ts — The ActivityStream is an EventEmitter with a circular buffer that receives real-time events from agent-loop, conversation-pipeline, memory, and collaboration modules. It now includes LlmStatsTracker (accessible via activityStream.llmStats) which subscribes to llm:end, llm:error, and llm:fallback events, maintaining per-bot aggregate counters: totalCalls, successCount, failCount, fallbackCount, avgDurationMs, lastCallAt, lastError, and a callerBreakdown (per-caller counts, durations, errors for conversation, planner, strategist, executor, feedback). Stats are in-memory only and reset on restart. The clearForBot() method also clears LLM stats. API endpoints at GET /api/agent-loop/llm-stats[/:botId].
