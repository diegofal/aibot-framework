Testing Guide

How to test each feature of the framework — automated tests, manual verification, and integration testing.

Running All Tests

bun test                    # Full suite (~3200 tests, ~48s)
bun test --watch            # Watch mode for development
bun test tests/some-file.ts # Run a specific test file

The test suite uses bun:test with vi mocks. One pre-existing failure in conversation-pipeline.test.ts (ToolDefinition format) is known and unrelated to new features.

1. Streaming Responses

Automated Tests

bun test tests/streaming.test.ts    # 15 tests

Covers: config schema validation, streamToChannel (progressive edits, throttling, single chunk, empty stream), streamToWebSocket (event sequence, closed connection), Ollama NDJSON parsing, LLMClient interface conformance.

Config

Streaming is enabled by default for all bots via conversation.streaming:

{
  "conversation": {
    "streaming": {
      "enabled": true,       // default: true — applies to all bots
      "editIntervalMs": 800, // throttle Telegram edits (200-5000ms)
      "minChunkChars": 50    // min chars before triggering an edit
    }
  }
}

Manual Testing

ChannelWhat happensHow to verify
Telegram Bot sends an initial message, then edits it progressively as tokens arrive. You see the response "typing out" in real-time. Send a message to a Telegram bot. The reply should appear and grow character by character. Check logs for "Using streaming path".
WebSocket (widget) Widget receives stream_start, multiple stream_chunk events, then stream_end with full text. Open the dashboard chat or embedded widget. Send a message. The response should appear progressively. Check browser DevTools WS frames.
REST API Tokens are collected internally, then returned as a single response (no streaming over HTTP). Call POST /api/v1/chat/:botId. Response arrives as a single JSON body.

When streaming is NOT used

  • Claude CLI backend (doesn't support streaming)
  • Tool-calling conversations (LLM needs structured responses)
  • Voice messages (TTS needs complete text)
  • When streaming.enabled: false in config

Troubleshooting

  • Streaming doesn't work on Telegram: Check that the Telegram channel adapter is providing _sendMessage and _editMessage handles. Check logs for "Streaming failed, falling back".
  • Messages appear in bursts: Increase editIntervalMs (Telegram rate-limits edits).
  • Streaming works but falls back: The LLM client must expose chatStream() — only OllamaLLMClient does. Claude CLI bots always use the non-streaming path.

2. Model Failover

Automated Tests

bun test tests/bot/model-failover/   # All failover tests

Covers: error classification (9 FailoverReasons), cooldown escalation, candidate chain traversal, backend-scoped skips, abort-chain logic.

Config

{
  "failover": {
    "enabled": true,
    "cooldownEnabled": true,
    "candidates": [
      { "backend": "ollama", "model": "qwen2.5:32b" },
      { "backend": "ollama", "model": "llama3.2:8b" },
      { "backend": "claude-cli" }
    ]
  }
}

If candidates is omitted, the chain is synthesized automatically from ollama.models.primary, ollama.models.fallbacks, and claudeCli.

Manual Testing

  1. Set failover.enabled: true in config/config.json
  2. Start the bot normally
  3. Stop Ollama: systemctl stop ollama
  4. Send a message to the bot — it should fall back to Claude CLI
  5. Check logs for "LLM failover: candidate 0 failed" and successful response from the fallback
  6. Restart Ollama — next request should use Ollama again (cooldown clears on success)

Error Classification Behavior

Error TypeFailoverReasonAction
HTTP 401/403authSkip ALL models on this backend
HTTP 402billingSkip ALL models on this backend, long cooldown (5h+)
HTTP 429rate_limitSkip this model, try next
HTTP 502/503/504, ECONNREFUSEDtimeoutSkip this model, try next
Context too longcontext_lengthAbort entire chain (no model will help)
HTTP 400, model not foundformatAbort entire chain (request is wrong)

3. Security Audit

Automated Tests

# Security check modules have unit tests in the security/ directory
bun test tests/bot/model-failover/   # Includes error classification used by audit

Config

{
  "security": {
    "auditOnStartup": true,  // default: true
    "cooldownMs": 86400000   // 24 hours between audits
  }
}

Manual Testing

  1. Start any bot with security.auditOnStartup: true (default)
  2. Check logs for one of:
    • "Security audit: clean" — no issues found
    • "Security audit: warnings found" — non-critical issues
    • "Security audit: CRITICAL issues found" — critical security issues
  3. Check the Activity page in the dashboard — a security:audit event should appear with findings
  4. The audit runs once per 24h per bot (cooldown file: .last-security-audit in soul dir)
  5. Delete the cooldown file to force a re-audit: rm data/tenants/__admin__/bots/<botId>/soul/.last-security-audit

What gets checked

  • Config secrets — API keys in config, .env file exposure
  • Filesystem permissions — World-writable directories, insecure file modes
  • Dangerous config flags — Risky configuration combinations
  • Model hygiene — Model configuration issues
  • Tool source scan (opt-in) — Dynamic tool code analysis

Manual Testing

To verify hooks fire, add a temporary listener in a skill or in bot-manager.ts after HookEmitter creation:

// In bot-manager.ts constructor, after this.hookEmitter = new HookEmitter():
this.hookEmitter.onHook('message_received', (e) => {
  console.log('HOOK: message_received', e.botId, e.text.slice(0, 50));
});
this.hookEmitter.onHook('after_tool_call', (e) => {
  console.log('HOOK: after_tool_call', e.toolName, e.success, e.durationMs + 'ms');
});

Then send messages and use tools — the console should show the hook events.

5. Memory Temporal Decay

Automated Tests

bun test tests/memory-temporal-decay.test.ts   # 22 tests

Covers: decay function (zero age, half-life, floor, future timestamps), MMR diversity reranking, config schema.

Config

{
  "soul": {
    "search": {
      "temporalDecay": {
        "enabled": true,      // default: true
        "halfLifeDays": 30,   // memories lose half their decay bonus after 30 days
        "weight": 0.3         // 30% of score is time-based, 70% is relevance
      },
      "mmr": {
        "enabled": false,     // Maximal Marginal Relevance for diversity
        "lambda": 0.7         // 1.0 = pure relevance, 0.0 = pure diversity
      }
    }
  }
}

Manual Testing

  1. Have a bot with existing memories spanning multiple days
  2. Use the memory_search tool to search for a topic
  3. Recent memories should rank higher than older ones with similar relevance
  4. Compare results with temporalDecay.enabled: false to see the difference

7. WhatsApp Integration

Automated Tests

bun test tests/whatsapp-integration.test.ts   # 26 tests

Covers: webhook signature verification, message extraction, whatsappToInbound conversion, image/interactive payload builders, status event parsing, channel creation.

Config

// In bots.json, add whatsapp config to a bot:
{
  "id": "my-bot",
  "name": "My Bot",
  "whatsapp": {
    "phoneNumberId": "123456789",      // From Meta Developer Console
    "accessToken": "EAAx...",          // Permanent access token
    "verifyToken": "my-verify-token",  // For webhook verification
    "appSecret": "abc123..."           // For webhook signature validation
  }
}

Setup

  1. Create a Meta/WhatsApp Business app at developers.facebook.com
  2. Get the Phone Number ID and permanent access token
  3. Set webhook URL to https://your-domain/api/whatsapp/webhook
  4. Subscribe to messages webhook field
  5. Configure the verify token to match your config

Manual Testing

FeatureHow to triggerWhat to verify
Text messages Send a WhatsApp message to your bot number Bot replies via WhatsApp. Check logs for "whatsapp message"
Image receiving Send a photo to the bot via WhatsApp Image URL appears in logs. Bot processes it if vision-capable model is used.
Interactive buttons (inline approval) Ask the bot to do something that requires a confirm-level tool (e.g. "write a file called test.txt"). The bot will ask for confirmation. WhatsApp shows approve/deny buttons. Tapping "Approve" executes the tool. Tapping "Deny" cancels it.
Image sending Bot generates or references an image URL in its response Image is sent via WhatsApp Cloud API media upload
Status tracking Send a message and wait Logs show "WhatsApp status" entries with delivered and read events

About Interactive Buttons (Inline Approval)

When the LLM calls a confirm-level tool (like file_write, exec, browser) in a WhatsApp conversation:

  1. The tool call is intercepted by InlineApprovalStore
  2. The LLM is told to ask the user for confirmation
  3. On WhatsApp, this appears as a message with two buttons: "Approve" and "Deny"
  4. The user's response (button tap or text "yes"/"no") is classified and the tool executes or is rejected

Note: The interactive buttons are only sent when the WhatsApp channel adapter detects a confirm-level tool. Regular text messages use normal replies.

8. Discord Channel

Automated Tests

bun test tests/discord-adapter.test.ts   # 19 tests

Covers: discordToInbound conversion (text, images, DMs, guilds), splitDiscordMessage (edge cases), channel construction, gateway event handling (Hello, READY, MESSAGE_CREATE, heartbeat, bot filtering, channel filtering).

Config

// In bots.json, add discord config to a bot:
{
  "id": "my-bot",
  "name": "My Bot",
  "discord": {
    "token": "MTIz...",                    // Bot token from Discord Developer Portal
    "applicationId": "123456789",          // Application ID (optional)
    "guildIds": ["987654321"],             // Guilds to listen in (optional, all if empty)
    "channelIds": ["111222333", "444555"]  // Channels to listen in (optional, all if empty)
  }
}

Setup

  1. Create a bot at discord.com/developers/applications
  2. Enable "Message Content Intent" in Bot settings (required to read message text)
  3. Generate a bot token
  4. Invite the bot to your server with bot and applications.commands scopes
  5. Add the config to your bot in bots.json

Manual Testing

  1. Add the Discord config to a bot and start the framework
  2. Check logs for "Discord gateway: ready" — confirms successful connection
  3. Send a message in a Discord channel the bot can see
  4. The bot should respond in the same channel
  5. Test long messages (>2000 chars) — they should be automatically split
  6. Test in DMs — bot should respond in DM too

Limitations

  • Uses Discord REST API + raw Gateway WebSocket (no discord.js dependency)
  • Outbound proactive messaging is not yet supported (requires cached channel IDs)
  • No slash commands — only message-based interaction
  • Auto-reconnects on disconnect with 5s delay

9. A2A Protocol (Agent-to-Agent)

Automated Tests

bun test tests/a2a/   # 62+ tests across 5 files

Covers: TaskStore CRUD/TTL/cancel, AgentCardBuilder, ToolAdapter, A2A types, AgentDirectory (heartbeat, pruning, search).

Config

{
  "a2a": {
    "enabled": true,              // Exposes ALL running bots as A2A agents
    "basePath": "/a2a",           // URL prefix
    "maxTasks": 1000,             // Max concurrent tasks in memory
    "taskTtlMs": 3600000,         // 1 hour TTL for completed tasks
    "agents": [                   // External A2A agents to connect to
      {
        "name": "external-agent",
        "url": "https://other-server/a2a/agent1",
        "timeout": 60000
      }
    ]
  }
}

Scope: A2A is a server-level setting — when enabled, it exposes all configured bots as A2A endpoints. Each bot gets its own /a2a/:botId route. External clients (other A2A servers, Claude Desktop, custom apps) can communicate with any bot through the protocol.

Manual Testing

Step 1: Enable and verify agent card

# Set a2a.enabled: true in config, restart
# Then fetch an agent card:
curl http://localhost:3000/a2a/my-bot/.well-known/agent.json | jq .

Should return:

{
  "name": "My Bot",
  "description": "Agent My Bot",
  "url": "http://127.0.0.1:3000/a2a/my-bot",
  "version": "1.0.0",
  "capabilities": { "streaming": false, "stateTransitionHistory": true },
  "skills": [...]
}

Step 2: Send a message

curl -X POST http://localhost:3000/a2a/my-bot \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "message/send",
    "params": {
      "message": {
        "role": "user",
        "parts": [{ "type": "text", "text": "Hello, who are you?" }]
      }
    },
    "id": "test-1"
  }' | jq .

Should return a task with status "completed" and the bot's response in messages.

Step 3: Get a task

curl -X POST http://localhost:3000/a2a/my-bot \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "tasks/get",
    "params": { "id": "TASK_ID_FROM_STEP_2" },
    "id": "test-2"
  }' | jq .

Step 4: Agent directory

# Register an external agent
curl -X POST http://localhost:3000/a2a/directory/register \
  -H "Content-Type: application/json" \
  -d '{"name": "test-agent", "url": "http://example.com/a2a/test", "skills": []}'

# List all agents
curl http://localhost:3000/a2a/directory/agents | jq .

# Search by skill
curl "http://localhost:3000/a2a/directory/agents?q=search" | jq .

# Heartbeat
curl -X POST http://localhost:3000/a2a/directory/heartbeat \
  -H "Content-Type: application/json" \
  -d '{"name": "test-agent"}'

12. Unified Error Classification

Automated Tests

bun test tests/bot/agent-retry-engine.test.ts   # 30+ tests

How it works

All LLM errors pass through classifyFailoverReason() which checks (in order):

  1. HTTP status code (most reliable)
  2. Node.js error code (ECONNREFUSED, ETIMEDOUT, etc.)
  3. Message regex patterns (last resort)

The result determines retry behavior in the agent loop AND failover decisions in FailoverLLMClient.
